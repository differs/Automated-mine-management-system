# Automated Mine Management System

[English Version](./README.en.md)

面向矿山运输场景的数字化调度产品基础仓库。

这个项目不是单纯做一个后台，而是要把矿山现场最核心的一条业务链路数字化：从调度派车，到司机到场排队，再到装车、称重、完单和数据沉淀，逐步替代微信群、电话和 Excel。

## 产品定位

这是一个服务矿山运输调度场景的业务系统，核心目标是解决三类问题：

- 调度效率低：派单靠电话和微信群，信息不同步
- 现场秩序差：坑口排队不可视，插队和扯皮频发
- 经营数据滞后：趟次、吨位、异常、效率都要靠人工汇总

一期先把“能跑、能控、能看”做出来。

- 能跑：派单、到场、排队、装车、称重、完单形成闭环
- 能控：关键节点留痕，人工干预可追溯
- 能看：调度室实时看板能看到车、坑、单、队列和异常

## 用户角色

- 调度员：看全局、派任务、处理异常
- 司机：收任务、到场打卡、看排队、看历史
- 坑口管理员：核验车辆、管理排队、确认装车
- 地磅操作员：录入重量、完成称重
- 财务 / 运营：查看产能、趟次、吨位和结算基础数据

## 一期范围

一期 MVP 聚焦最短业务闭环，不追求功能大而全。

- 调度后台：Web 管理端
- 司机移动端：Flutter App
- 坑口作业端：Flutter App 或轻量 H5
- 实时排队：Redis
- 业务主库：PostgreSQL
- 实时消息：WebSocket
- 基础看板：Vite 前端大屏/后台图表

## 技术路线

- 后端：Rust
- 前端：Vite
- 手机端：Flutter
- 数据库：PostgreSQL
- 缓存与队列状态：Redis
- 部署：Docker + Nginx

后端建议优先采用 Rust Web 框架组合：

- `axum`：HTTP API 和 WebSocket
- `sqlx`：数据库访问和迁移
- `tokio`：异步运行时
- `redis`：实时队列和状态缓存
- `serde`：序列化

## 第一阶段目标

第一阶段不是追求复杂算法，而是把现场调度标准化。

1. 让每一张运单都有状态、有责任人、有时间线
2. 让每一个坑口的排队情况可视、可控、可追溯
3. 让每一天的趟次、吨位、异常和效率自动沉淀

## 后续方向

- 对接车牌识别和摄像头
- 对接地磅自动采集
- 做司机推荐和坑口分流
- 做异常识别和拥堵预测
- 打通产量、销售、结算链路
