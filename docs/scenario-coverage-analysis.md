# 真实场景覆盖度分析

## 结论

目前方案没有完全覆盖矿山真实场景，但已经覆盖了最核心的“运输调度主链路”。

如果目标是一期替换微信群、电话和 Excel 的主流程，当前方案是够用的。

如果目标是“完全还原并接管矿山现场全部复杂操作”，当前方案还差一批关键边界场景，需要在一期试运行前补齐规则，在二期继续产品化。

## 已覆盖的核心场景

### 1. 调度派车

- 调度员创建运单
- 指派司机和坑口
- 追踪运单状态

这部分已经覆盖矿山运输最核心的任务分发场景。

### 2. 到场排队

- 司机到场登记
- 自动进入队列
- 坑口管理员查看顺序和叫号

这部分已经覆盖现场秩序管理的主问题，也能初步解决插队争议。

### 3. 装车与称重

- 坑口开始装车、结束装车
- 地磅录入重量
- 运单完成闭环

这部分覆盖了生产链路中的关键结果数据。

### 4. 实时看板

- 今日派单
- 完成趟次
- 排队时长
- 吞吐量
- 异常提醒

这部分覆盖了管理层和调度员最基础的实时可视化需求。

## 当前未完全覆盖的真实场景

### 1. 多装点 / 多卸点 / 复合运输线路

当前模型偏向“一个运单对应一个坑口”的简单模式。

真实矿山经常出现：

- 同一司机今天在多个坑口来回跑
- 装车点和卸货点都需要记录
- 同一趟存在中转点或不同线路价格

这意味着后续要补：

- 卸点管理
- 线路管理
- 运价规则管理

### 2. 班次和交接班

很多矿山不是自然日管理，而是白班、夜班分开统计。

当前文档对以下问题还没定：

- 班次切换时队列怎么继承
- 夜班跨天运单算哪一天
- 班组负责人和交接记录怎么留痕

### 3. 临时插单和领导特批

现场一定会有临时插队、急单、领导特批车。

系统不能假设现场永远按理想顺序运行。

需要补的不是“禁止插单”，而是：

- 允许授权插单
- 强制记录插单原因
- 在报表里把插单作为异常管理项

### 4. 断网、弱网和离线补录

我们提到了弱网，但现在还没有细化到真实操作级别。

比如：

- 司机到场时完全没网怎么办
- 坑口端断网时如何继续叫号
- 称重后网络中断如何避免重复提交

这些都需要形成明确的离线策略和补录机制。

### 5. 地磅业务细节

称重不是简单输入一个净重就结束。

现场可能存在：

- 固定皮重和实时皮重并存
- 一车两过磅
- 地磅设备故障，临时手工记录
- 重量修正和复磅

当前 DDL 还是一期简化版，后续要增强称重模型。

### 6. 外协车队和司机身份复杂性

矿山常见自有车、外协车、挂靠车混用。

真实情况里经常会有：

- 一个人开不同车
- 一辆车多人轮班
- 司机手机号不稳定
- 司机并不总是自己操作手机

当前司机模型是一人一车的简化设计，适合 MVP，但不等于真实全貌。

### 7. 安全与合规动作

真实矿山现场常常会有这些要求：

- 司机进场安全确认
- 黑名单车辆禁入
- 超载预警
- 证件有效期提醒

这些在当前方案中还没有明确产品入口。

### 8. 财务结算链路

当前只覆盖了结算基础数据，没有覆盖真正的结算业务。

现实里结算可能按：

- 吨位
- 趟次
- 线路
- 坑口
- 车型
- 时间段

如果一期只做调度，这是合理的；但如果业务方期待系统直接替代结算 Excel，就还不够。

### 9. 多矿区 / 多公司权限隔离

如果未来不是只服务单一矿区，而是多个矿区、多承运主体并行，当前权限模型不够细。

后续需要考虑：

- 租户隔离
- 矿区隔离
- 数据域权限

## 一期是否足够

如果一期目标是以下范围，当前方案基本成立：

- 先在一个矿区或一个坑口试运行
- 先让调度流程在线化
- 先拿到真实运单、队列和称重数据
- 先跑出稳定的日报和实时看板

如果一期目标被抬高到下面这个标准，当前方案就不够：

- 一上线就覆盖所有矿区
- 一次性替代所有 Excel 和纸质流程
- 一次性接管复杂结算和所有例外场景

这不是技术问题，而是产品分期问题。

## 建议补充项

建议在下一轮需求澄清中优先补这 6 项：

1. 班次模型和统计口径
2. 卸点、线路和运价规则
3. 插单、特批和人工调整权限
4. 离线策略和补录机制
5. 地磅复磅、更正和设备故障流程
6. 司机、车辆、车队之间的真实关系模型
