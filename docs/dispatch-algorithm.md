# 矿山运输调度算法说明

> ⚠️ **重要前提**：矿山调度算法不能通用化。每个矿山的坑口布局、地形坡度、道路条件、装载点位置都不一样。以下算法是基础框架，**实际部署时必须根据矿山现场情况调参和优化**。

---

## 一、算法的前提：地形决定调度策略

### 不同矿山，调度逻辑完全不同

```
露天煤矿（内蒙古）          露天金属矿（云南）          地下矿（山东）
─────────────────          ────────────────          ────────────────
- 坑口集中，道路平坦       - 坑口分散，山区坡道      - 竖井/斜井运输
- 卡车载重60-100吨        - 卡车载重30-50吨         - 电机车/罐笼
- 主要优化：减少排队       - 主要优化：路线选择      - 主要优化：提升效率
- 算法侧重：负载均衡       - 算法侧重：距离计算      - 算法侧重：时序安排
```

### 部署时必须采集的现场数据

```
每个坑口部署前，需要现场勘察确认：
  □ 坑口坐标（GPS取点，用于距离计算）
  □ 坑口到地磅/堆场的道路距离（实测，不是直线）
  □ 道路坡度（影响重车上坡速度）
  □ 坑口装载能力（同时能装几辆车）
  □ 地磅通过能力（每小时能称重几辆）
  □ 典型运输路线（A坑→1号破碎站，B坑→2号堆场）
  □ 每趟平均时间和距离
```

## 二、算法架构：基础框架 + 地形适配层

```
调度算法
├── 基础框架（通用，不变）
│   ├── 运单生命周期管理
│   ├── 排队管理（FIFO + 叫号）
│   ├── 状态流转（到场→排队→装车→称重→完单）
│   └── 数据采集（位置、时间、重量）
│
└── 地形适配层（每个矿山不同）← 核心优化点
    ├── 距离矩阵（坑口到坑口、坑口到地磅的实际路径距离）
    ├── 道路权重（坡度、路面条件、拥堵时段）
    ├── 坑口优先级（哪个坑口优先调度）
    └── 车辆适配（什么车适合什么路线）
```

### 2.1 通用基础框架

```
派单逻辑（通用，不依赖地形）：
  1. 司机A完成当前运单 → 状态变为"空闲"
  2. 系统查找该司机当前位置（关联最近一次位置上报）
  3. 系统查找所有待派运单 → 计算最优匹配
  4. 推荐派单方案 → 调度员确认

排队逻辑（通用）：
  1. 司机到场 → 入队（记录时间）
  2. 按入队顺序叫号（FIFO）
  3. 异常情况可插队（需授权）
```

### 2.2 地形适配层（每个矿山单独配置）

```
距离矩阵（示例）：
             坑口A    坑口B    坑口C    地磅1    堆场1
坑口A          0       2.3km    4.1km    1.5km    3.2km
坑口B        2.3km      0       1.8km    2.8km    4.5km
坑口C        4.1km    1.8km      0       3.5km    5.2km
地磅1        1.5km    2.8km    3.5km      0       1.2km

坑口参数（示例）：
          装载能力    同时装载数    优先级    道路条件
坑口A      5分/车       2辆         高        硬化路
坑口B      8分/车       1辆         中        土路/坡道
坑口C      6分/车       2辆         低        碎石路

车辆适配（示例）：
          适合路线    载重    油耗
卡车#01   坑口A/B     60t     高
卡车#02   坑口B/C     45t     低
卡车#03   全部        80t     高
```

## 三、纯算法模式：加权优先级匹配算法（WPMA）

### 3.1 基础权重公式

```
综合权重 = w1 × 空闲权重 + w2 × 工作量权重 + w3 × 距离权重 + w4 × 坑口权重

其中：
  空闲权重 = idle_minutes / max_idle    公平性（休息越久越优先）
  工作量权重 = 1 - (trips_today / max_trips)  负荷均衡（趟次越少越优先）
  距离权重 = 1 - (distance_to_pit / max_distance)  效率（距离越近越优先）
  坑口权重 = pit_priority / max_priority   业务优先级

  默认权重：w1=0.3, w2=0.2, w3=0.3, w4=0.2
  （可调，不同矿山侧重点不同）
```

### 3.2 地形因素如何影响权重

```
平坦矿山的算法侧重                   山区矿山的算法侧重
────────────────                     ────────────────
w3（距离权重）= 0.4  ← 更高          w3（距离权重）= 0.5  ← 更高
w1（空闲权重）= 0.3                   w1（空闲权重）= 0.2  ← 更低
w2（工作量权重）= 0.2                 w2（工作量权重）= 0.2
w4（坑口权重）= 0.1                   w4（坑口权重）= 0.1

理由：平坦矿山跑得快，距离影响大        理由：山区道路耗时差距大
      空闲时间不那么重要                       远距离空跑成本高
```

### 3.3 算法参数必须现场调优

```
部署到新矿山时的调优流程：
  1. 先用默认参数跑1周，收集数据
  2. 分析瓶颈：
     - 排队太长？→ 降低距离权重，提高坑口权重
     - 空跑太多？→ 提高距离权重
     - 司机不公平？→ 提高空闲权重
  3. 调整权重参数，观察1-2天
  4. 重复2-3步，直到满意
  
  每次新矿山部署都需要这个流程。
  没有一套参数适用于所有矿山。
```

## 四、AI模式的算法（盘古大模型）

AI模式不是替代基础算法，而是在关键决策点优化：

```
AI做的事：
  - 分析历史数据规律（哪些时段堵、哪个坑口效率低）
  - 根据实时状态推荐最优派单（超越固定公式）
  - 发现人工调度中的隐性规则（老调度员的经验）

AI不做的事：
  - 不接管基础流程（运单流转、排队、状态管理还是代码控制）
  - 不做实时控制（刹车、油门、方向盘）
  - 不替代调度员的最终决策权
```

### AI在地形适配上的优势

```
固定算法：距离权重固定，不能适应动态变化
AI算法：盘古分析各条路线的实时耗时，动态调整推荐

例如：
  上午9点，坑口A到地磅的路正在修，堵车
  固定算法：仍然按3分钟计算距离权重
  AI算法：识别到修路堵车，自动推荐司机去坑口B
```

## 五、部署流程（每个矿山都要做）

```
第一步：现场勘察
  □ 用GPS记录所有坑口、地磅、堆场的坐标
  □ 测量各点之间的实际行车距离（不是直线距离）
  □ 记录道路条件（硬化路/土路/坡道）
  □ 记录坑口装载能力、地磅通过能力
  
第二步：配置参数
  □ 输入距离矩阵
  □ 设置坑口优先级
  □ 设置权重参数（w1-w4）
  □ 设置阈值（拥堵预警线、超时时间）
  
第三步：试运行
  □ 纯算法模式跑1周，验证基础流程
  □ 收集数据，分析瓶颈
  □ 调参优化
  
第四步：可选升级
  □ 客户有意愿 → 开启AI模式
  □ 盘古API接入 → AI辅助调度
```

## 六、算法适配清单（给客户的问卷）

```
部署前请提供以下信息：
  1. 矿区地图（坑口、地磅、堆场位置）
  2. 各点之间的道路距离（不是直线距离，是车走的路线）
  3. 道路条件（硬化路/土路/坡度大/窄路）
  4. 每个坑口的装载速度（分钟/车）
  5. 地磅的通过能力（辆/小时）
  6. 常用运输路线（哪个坑口对应哪个堆场）
  7. 高峰时段（哪些时间车辆最集中）
  8. 典型运输距离（近/中/远）
```

## 七、结论

```
算法 = 通用框架（60%）+ 地形适配（40%）

交付系统 ≠ 部署算法
每个矿山都需要单独调参，没有"一套走天下"的调度算法。
这才是做矿山调度系统的核心壁垒——不是代码，是对现场的理解。
```
