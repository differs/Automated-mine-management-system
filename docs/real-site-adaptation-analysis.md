# 实际矿山场景适配分析

基于对天弘矿业集团的全面调查，分析我们的 auto-mining-system 是否能满足其真实需求。

---

## 一、天弘矿业的真实矿山分布

### 矿山位置（非单一矿区）

```
贵州沿河县 ──── 核心矿区（7宗矿山）
  黄土矿、中寨矿、后坪矿等
  矿石运输 → 洪渡镇产业园（206亩自建工厂）

贵州务川县 ──── 扩展矿区
  持有矿权

重庆黔江区 ──── 跨省矿区
  四季采矿场（持有矿权，已在重庆规自局备案）

重庆彭水县 ──── 加工基地
  有重晶石磨粉加工厂

广西防城港 ──── 沿海加工+出口基地
  广西天弘矿业（20人），年产12万吨重晶石矿粉

广西来宾象州 ──── 加工基地
  广西弘象矿业

新疆轮台 ──── 西部加工基地
  贴近中石油西部油田客户

广东佛山 ──── 加工基地
  高消光钡生产线
```

### 物流链条（非短倒运输）

```
矿山（贵州/重庆山里）
    ↓ 盘山公路运输（30-100km）
加工厂/选矿厂（洪渡/防城港/彭水等）
    ↓ 国道/高速运输（数百至数千公里）
中石油/中石化/中海油 油田现场
  OR
出口港口（防城港 → 美国/澳大利亚）
```

**这和系统当前假设的"矿山内部短倒运输"有本质区别。**

## 二、关键差异分析

### 差异 1：运输模式不同

| 维度 | 系统当前假设 | 天弘实际情况 |
|------|------------|------------|
| 运输类型 | 矿山内部短倒（坑口→堆场/地磅） | 矿山→工厂（长途）+ 工厂→客户（全国物流） |
| 运输距离 | 几百米到几公里 | 几十到几千公里 |
| 司机管理 | 自有司机，单车单人 | 7000名司机，含外协车队、挂靠车 |
| 派单方式 | 调度员派单给指定司机 | 可能涉及车队接单→分派司机 |
| 趟次频率 | 每天50+趟（高频） | 每天50+趟但涉及不同路线 |

**影响：**
- ⚠️ 当前运单模型是"一个坑口→一个地磅"的短倒模式
- ⚠️ 实际上需要支持"多装点→多卸点"的长途运输模式
- ⚠️ 需要支持按路线、按吨位、按距离的计费模型

### 差异 2：多地点、分布式

```
矿山 A（沿河） ──┬──→ 工厂 X（洪渡）
矿山 B（务川） ──┤
矿山 C（黔江） ──┼──→ 工厂 Y（防城港）
                  │
                  └──→ 工厂 Z（新疆轮台）
```

**系统需要的能力：**
- 🔴 **多矿区数据隔离** — 每个矿区的运营数据需要分开管理
- 🔴 **分布式/离线架构** — 山区网络差，需要本地缓存+异步同步
- 🟡 **多级权限** — 集团总部能看到全局，各矿区只能看自己的
- 🟡 **统一的车辆/司机管理** — 司机可能跨矿区作业

### 差异 3：网络环境

| 地点 | 网络条件 | 问题 |
|------|---------|------|
| 广州/成都办公室 | ✅ 光纤宽带 | 无问题 |
| 洪渡镇产业园 | ⚠️ 乡镇4G/宽带 | 可能不稳定 |
| 沿河矿山（山里） | ❌ 弱网/无信号 | **严重问题** |
| 重庆黔江矿山 | ❌ 山区弱网 | **严重问题** |
| 新疆轮台 | ⚠️ 偏远地区 | 可能不稳定 |

**结论：离线能力不是"可选功能"，而是**必需功能**。**

### 差异 4：多实体、多公司

| 实体 | 角色 | 系统需求 |
|------|------|---------|
| 贵州天弘矿业 | 核心矿业公司 | 主数据、生产管理 |
| 池州天弘矿业 | 贸易公司 | 销售、结算 |
| 广西天弘矿业 | 加工基地 | 生产管理 |
| 广西弘象矿业 | 矿业公司 | 独立运营 |
| 新疆子公司 | 加工基地 | 独立运营 |
| 广州刻雅馆/GlobalX | 医药/出口 | 可能无关 |

**当前系统是单租户架构，不支持多公司隔离。**

### 差异 5：硬件对接需求

```
矿山现场：
  地磅 ─── RS232/串口 ─── 需要自动读取重量
  摄像头 ─── RTSP流 ─── 需要车牌识别
  GPS ─── 司机位置追踪
  道闸 ─── 自动抬杆放行

当前系统状态：
  ❌ 地磅：仅支持手动录入（API有source字段但无串口驱动）
  ❌ 车牌识别：未实现
  ❌ GPS追踪：未实现
  ❌ 道闸控制：未实现
```

## 四、需要改造才能满足的部分

### 🔴 必须改造（不上线跑不了）

| 改造项 | 说明 | 工作量 |
|--------|------|--------|
| **多矿区/多租户** | 数据库加 tenant_id，API按租户隔离 | **大**（架构级） |
| **离线能力** | 本地缓存、操作队列、异步同步、冲突解决 | **大**（架构级） |
| **长途运输路线** | 运单支持起点→终点多卸点，非单坑口模式 | **中** |
| **车队管理** | 外协车队接单→分派给司机的流程 | **中** |
| **结算模型** | 按路线/吨位/车型/距离的多维计价 | **中** |

### 🟡 重要但可后续迭代

| 改造项 | 说明 | 工作量 |
|--------|------|--------|
| **地磅串口对接** | 支持主流地磅协议自动读取重量 | **中** |
| **车牌识别** | 对接摄像头或第三方OCR服务 | **中** |
| **GPS定位** | 司机签到时的位置校验 | **小** |
| **多语言/国际化** | 巴西团队可能需要葡语界面 | **中** |
| **移动端完善** | 从H5升级到Flutter App，增强离线能力 | **大** |

## 五、实际需要的特殊配置

### 1. 网络架构建议

```
                    ┌──────────────┐
                    │   阿里云/腾讯云  │
                    │  (主服务器)    │
                    │ PostgreSQL    │
                    │ Redis         │
                    │ API           │
                    │ Admin Web     │
                    └──────┬───────┘
                           │
          ┌────────────────┼────────────────┐
          │                │                │
    ┌─────▼─────┐   ┌─────▼─────┐   ┌─────▼─────┐
    │ 洪渡产业园  │   │ 沿河矿山   │   │ 新疆轮台   │
    │ (4G+宽带)  │   │ (弱网)    │   │ (4G)      │
    │ Web直连云端 │   │ 本地边缘   │   │ Web直连    │
    │            │   │ 服务器+缓存 │   │           │
    └────────────┘   └───────────┘   └───────────┘
```

- 沿河、黔江等弱网矿山需要部署**本地边缘服务器**（一台MiniPC或工控机即可）
- 边缘服务器运行本地API + SQLite缓存，网络恢复后同步到云端
- 司机端App需要支持**离线操作**（任务缓存、到场记录本地保存）

### 2. 司机端特殊需求

7000名司机中很多是**文化程度不高、不擅长用智能手机**的群体：

```
需要的能力：
  ✅ 极简界面 — 大字、图标化、尽量少文字
  ✅ 语音播报 — 叫号时语音提醒"张三，到1号坑口装车"
  ✅ 代操作 — 坑口管理员可以替司机操作（当前系统已设计此能力）
  ✅ 微信集成 — 很多矿工用微信比用App熟练，小程序是更好的入口
  ❌ 无需注册 — 直接用手机号+验证码登录，不要复杂的注册流程
```

### 3. 多矿山的调度规则

| 规则 | 说明 |
|------|------|
| 司机跨矿山调度 | 同一个司机可能今天在沿河拉货，明天在务川 |
| 车辆跨工厂调度 | 车队的车可能去不同工厂装不同矿石 |
| 优先级规则 | 中石油的急单需要插队处理 |
| 品控要求 | 不同客户对品级要求不同，需要匹配矿源 |
| 重量地域差异 | 不同省份地磅可能有不同的校准标准 |

## 七、建议的实施路径

### Phase 0：单矿区试点（与招聘JD吻合）

用当前系统直接在洪渡产业园试运行：
- 覆盖产业园内部运送（坑口→地磅）
- 验证核心流程
- 收集现场反馈
- **当前系统85%可用，改造量最小**

### Phase 1：多矿区扩展

- 加租户/矿区隔离
- 加离线同步
- 加路线/卸点模型

### Phase 2：全集团推广

- 多公司数据隔离
- 结算系统
- 硬件对接（地磅/摄像头）
- 移动端优化

## 八、总结

**我们的系统能不能满足天弘的实际需求？**

> **能，但需要分阶段。**
>
> 如果他们的第一步确实是"先在一个矿区跑通调度流程"，那我们的系统几乎是量身定做的——从数据库设计到API到前端，覆盖了煤矿/金属矿内部短倒运输的全流程。
>
> 但如果他们一上来就要覆盖"贵州7个矿山+广西加工厂+新疆基地+7000名司机长途运输"，那当前系统还有较大差距，主要缺多租户、离线能力和物流路线模型。
>
> 好消息是：**招聘JD描述的首要项目正是"搭建矿山运输车辆调度系统"——这正是我们系统当前做得最好的部分。**
